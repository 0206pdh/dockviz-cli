## Problem Statement

dockviz-cli is a Docker TUI written in Go. The core question: why would a DevOps/infra
engineer use dockviz over lazydocker (40k+ stars)?

Currently there is no terminal-based tool that shows Docker network topology interactively
in real time. All network visualization tools require a web browser (DockerNet, Atlas) or
generate static SVG files (docker-network-graph). Engineers who manage servers via SSH have
no interactive network topology view.

## What Makes This Cool

"장애가 터졌을 때 dockviz를 열면, 뭐가 왜 죽었는지 바로 보인다."

A container crashes at 3am. The on-call engineer SSHs in. They type `dockviz`. They
immediately see:
- The network topology with a red node (the crashed container)
- The event timeline showing the crash event and its cascading effects
- Which other containers on the same network were affected

lazydocker shows logs and stats. dockviz shows the STORY of what happened.

## Constraints

- Terminal-only (no browser dependency)
- Go + Bubble Tea/tview TUI framework
- Must work over SSH (no GUI calls)
- Single binary distribution (existing CI for arm64/amd64 Linux/macOS/Windows)

## Premises

1. dockviz의 핵심 차별점은 "TUI 안에서 Docker 네트워크를 실시간으로 시각화"다
   — lazydocker/Oxker가 못 하는 것.
2. 주요 타겟은 SSH로 서버를 관리하는 DevOps/인프라 엔지니어다.
3. 지금 당장 만들어야 할 건 Compose 시각화/멀티호스트가 아니라 현재 브랜치의
   네트워크 토폴로지 + 이벤트 타임라인을 제대로 완성하는 것이다.

## Approaches Considered

### Approach A: 토글 오버레이 (Minimal Viable)
기존 컨테이너 뷰에 토폴로지/이벤트 패널을 단축키로 토글.
Effort: S | Risk: Low | Completeness: 6/10
Con: "lazydocker에 탭 하나 추가한 것"으로 인식될 위험.

### Approach B: 네트워크 퍼스트 레이아웃 (Ideal Architecture)
토폴로지 그래프가 메인 뷰. 노드 선택 → 우측 패널에 로그/stats.
Effort: L | Risk: High | Completeness: 9/10
Con: 기존 UX 전면 재설계, ASCII 그래프 복잡도 한계.

### Approach C: 장애 전파 시각화 (Selected)
컨테이너 크래시/재시작 → 토폴로지에서 빨간 노드 강조 + 이벤트 타임라인에
연쇄 재시작 체인 자동 표시. DevOps 엔지니어의 실제 고통점 해결.
Effort: M | Risk: Medium | Completeness: 8/10

## Recommended Approach

**Approach C: 장애 전파 시각화**

### Core UX

아래는 **Networks 탭 내부** 레이아웃 (탭 바는 기존 유지):

```
┌─────────────────────────────────────────────────────┐
│ dockviz v0.x  [Containers] [Networks*] [Images] [Events] │
├──────────────────────┬──────────────────────────────┤
│ NETWORK TOPOLOGY     │ EVENT TIMELINE               │
│  (Networks 탭 내부)   │                              │
│  [nginx] ──── web    │ 03:21:44 container/die       │
│     │                │   api-server (OOMKilled,     │
│  ✗ api  DEAD         │   exit=137)                  │
│     │                │ 03:21:45 container/restart   │
│  ● db   UP           │   api-server                 │
│                      │ 03:22:01 container/die       │
│  ✗=dead ◑=restart    │   api-server                 │
│  ●=running           │                              │
├──────────────────────┴──────────────────────────────┤
│ [tab] Switch panel  [↑↓] Scroll  [?] Help           │
└─────────────────────────────────────────────────────┘
```

### Layout: Networks 탭 안에 스플릿 뷰

기존 4-탭 구조(Containers / Networks / Images / Events)는 유지.
Networks 탭 내부를 좌(토폴로지)/우(이벤트 타임라인) 스플릿으로 교체.

```
탭 바: [Containers] [Networks*] [Images] [Events]
                      ↑ 이 탭 내부가 스플릿으로 바뀜
```

모든 기존 기능(로그 스트리밍, 컨테이너 삭제, sparkline 등)은 Containers 탭에
그대로 유지됨. 완전히 additive.

---

### 패키지 구조 변경 (Import Cycle 방지)

**하드 블로커:** `tui` 패키지가 이미 `ui`를 import함. `ui.RenderNetworkGraph`가
`tui.ContainerState`를 받으면 import cycle → 컴파일 불가.

**해결:** `ContainerState`를 `internal/docker` 패키지로 이동.
`ui`와 `tui` 모두 `docker`를 이미 import하고 있으므로 cycle 없음.

```
internal/docker/state.go  ← 새 파일 (ContainerState 정의)
internal/ui/graph.go      ← docker.ContainerState를 파라미터로 받음
internal/tui/model.go     ← docker.ContainerState 맵을 필드로 가짐
```

---

### 이벤트 스트리밍 시작 시점 변경

**현재 문제:** 이벤트 스트리밍이 Events 탭 최초 방문 시에만 시작됨
(`update.go:188`). Networks 탭에서 토폴로지 상태를 보여주려면
앱 시작 시점에 스트리밍을 시작해야 함.

**변경:** `newModel()` 또는 `Init()`에서 `waitForEventCmd()`를
무조건 시작. 기존 Events 탭 방문 시 중복 시작 로직 제거.
`eventMsg` 핸들러는 두 용도로 동시에 작동:
1. `m.events` 슬라이스 갱신 (타임라인 패널)
2. `m.containerStates` 맵 갱신 (토폴로지 오버레이)

---

### 데이터 모델

```go
// internal/docker/state.go (새 파일)
type ContainerState struct {
    Status       string    // "running" | "dead" | "restarting"
    ExitCode     int       // 0=정상, 137=SIGKILL, 1=에러
    OOMKilled    bool      // container/die 의 oomKilled Attribute
    RestartCount int       // 세션 내 재시작 횟수 (앱 재시작 시 리셋됨 — 알려진 한계)
    UpdatedAt    time.Time
}

// internal/tui/model.go — Model 구조체에 추가
ContainerStates map[string]docker.ContainerState // key = ContainerName (not ID)
```

**key를 ContainerName으로 쓰는 이유:** `NetworkInfo.Containers`는 이름만 노출함.
ContainerID로 키잉하려면 name→ID 역매핑이 필요한데 불필요한 복잡도.
같은 이름의 컨테이너가 여러 네트워크에 있어도 동일 컨테이너이므로 name은 충분.

**stale 상태 정리:** `container/destroy` 이벤트 수신 시 `ContainerStates`에서
해당 항목 삭제. `container/start` 이벤트 시 `RestartCount` 리셋.

**중요 — Bubble Tea 단일 고루틴 제약:** `ContainerStates`는 반드시
`Update()` 루프 안에서만 읽고 씀. 배경 고루틴에서 직접 쓰지 말 것.

---

### Docker 이벤트 → 상태 매핑

**주의:** `container/oom`은 실제 Docker 이벤트 Action이 아님.
OOM 정보는 `container/die` 이벤트의 Attributes에서 가져옴:

```go
// internal/docker/events.go — EventInfo 구조체에 필드 추가
type EventInfo struct {
    Time          time.Time
    Action        string
    ContainerName string
    ContainerID   string
    ExitCode      int    // msg.Actor.Attributes["exitCode"] 파싱
    OOMKilled     bool   // msg.Actor.Attributes["oomKilled"] == "true"
    // RestartCount는 EventInfo에 없음 — ContainerState에서 누적 관리
}

// 이벤트 → ContainerState 전이 테이블
// container/die     → Status="dead",        OOMKilled=..., ExitCode=...
// container/start   → Status="running",     RestartCount=0
// container/restart → Status="restarting",  RestartCount++
// container/destroy → ContainerStates에서 항목 삭제
// container/kill    → 상태 변경 없음 (die 이벤트가 곧 따라옴)
```

---

### graph.go 리팩터링 (현재: 플랫 텍스트 렌더러)

현재 `RenderNetworkGraph`는 네트워크별로 한 줄씩 텍스트를 출력하는
플랫 렌더러. 노드 개념이 없어서 상태 오버레이를 바로 붙일 수 없음.

**변경 방향:**

```go
// 새 시그니처 (docker.ContainerState — import cycle 없음)
func RenderNetworkGraph(
    networks []docker.NetworkInfo,
    states   map[string]docker.ContainerState, // key = ContainerName
) string

// 각 컨테이너 이름 렌더링 시:
func containerLabel(name string, states map[string]docker.ContainerState) string {
    s, ok := states[name]
    if !ok {
        return "● " + name // 상태 미확인 = healthy로 표시
    }
    switch s.Status {
    case "dead":
        return lipgloss.NewStyle().Foreground(lipgloss.Color("9")).Render("✗ " + name)
    case "restarting":
        return lipgloss.NewStyle().Foreground(lipgloss.Color("11")).Render("◑ " + name)
    default:
        return lipgloss.NewStyle().Foreground(lipgloss.Color("10")).Render("● " + name)
    }
}
```

**lipgloss fallback 주의:** `graph.go`는 현재 TUI 외 경로에서도 호출될 수 있음.
ANSI 코드가 파이프/리다이렉트를 오염시키지 않도록 `lipgloss.HasDarkBackground()`
또는 `isatty` 체크 후 plain-text fallback 분기 추가.

**스플릿 레이아웃 (view.go):**
```go
leftWidth  := m.width / 2
rightWidth := m.width - leftWidth - 1 // 구분선 1칸

// 최소 너비 80 미만이면 수직 스택으로 fallback
if m.width < 80 {
    return lipgloss.JoinVertical(lipgloss.Left, topologyView, timelineView)
}
return lipgloss.JoinHorizontal(lipgloss.Top,
    lipgloss.NewStyle().Width(leftWidth).Render(topologyView),
    "│",
    lipgloss.NewStyle().Width(rightWidth).Render(timelineView),
)
```

컨테이너 수가 많을 때(15+)는 네트워크 수 내림차순으로 정렬해 상위 15개만 표시하고
"... and N more" 추가. "네트워크 수" = 해당 컨테이너가 속한 NetworkInfo 항목 수.

### Implementation Plan

**Phase 1: ContainerState 모델 + 이벤트 → 상태 전이**

1. `model.go` — `ContainerState` 구조체 + `ContainerStates map[string]ContainerState`
2. `events.go` — `EventInfo`에 `ExitCode`, `OOMKilled`, `RestartCount` 추가;
   `msg.Actor.Attributes` 파싱 (container/die, container/start, container/restart만)
3. `update.go` — `eventMsg` 핸들러에서 `ContainerStates` 맵 갱신

**Phase 2: graph.go에 상태 오버레이**

1. `RenderNetworkGraph` 시그니처에 `states map[string]ContainerState` 추가
2. `containerLabel()` 헬퍼로 상태별 ANSI 색 + 아이콘 적용
3. `view.go` — Networks 탭을 좌/우 스플릿으로 변경; 좌=토폴로지, 우=타임라인

**Phase 3: 이벤트 타임라인 패널**

1. `view.go` 우측 패널에 이벤트 리스트 렌더링
   - 기존 `m.events` 슬라이스 재사용 (이미 최대 100개 캡)
   - 최신 이벤트가 위로 오는 역순 정렬
2. 이벤트 타입별 색상: die=red, restart=yellow, start=green
3. 표시 포맷: `HH:MM:SS [container/die] api-server (OOMKilled, exit=137)`

**Phase 4 (v0.3.0으로 연기):**
- 연쇄 이벤트 체인 그룹화
- `--demo` 크래시 시나리오 시뮬레이션

`eventMsg` 핸들러가 시작부터 `ContainerStates`를 갱신하므로,
앱 실행 전에 죽어있던 컨테이너도 1시간 이내 사건은 복원됨. 채널은 순서 보장.

## Success Criteria

- `docker kill <container>` 실행 시 Networks 탭 토폴로지에서 해당 노드가
  빨간색 ✗ 로 바뀌고 우측 타임라인에 die 이벤트가 표시됨
  (이벤트 채널은 실시간이므로 즉각 반영; 토폴로지는 이벤트 수신 시점에 갱신)
- docker-compose up 환경에서 네트워크 관계가 정확하게 그래프로 표시됨
- "이걸 lazydocker로는 할 수 없다"는 게 스크린샷 한 장으로 설명됨
