# 컨테이너 상태, 네트워크 드라이버 & 이벤트 액션 레퍼런스

dockviz에 표시되는 모든 상태, 드라이버, 이벤트 액션에 대한 설명입니다.

---

## 컨테이너 상태

| 상태 | 아이콘 | 색상 | 의미 |
|------|--------|------|------|
| `running` | ● | 초록 | 컨테이너 프로세스가 실행 중입니다. |
| `exited` | ○ | 빨강 | 프로세스가 종료됐습니다. 파일시스템은 `docker rm`으로 삭제하기 전까지 보존됩니다. |
| `paused` | ◑ | 노랑 | 컨테이너의 모든 프로세스가 `SIGSTOP`으로 일시 정지됐습니다. CPU 사용량은 0이 되지만 메모리는 그대로 유지됩니다. `docker unpause`로 재개할 수 있습니다. |
| `restarting` | ↻ | 파랑 | 컨테이너가 충돌하거나 중지된 후 재시작 정책에 의해 재기동 중입니다. `die` 이벤트와 다음 `start` 이벤트 사이에 잠깐 나타납니다. |
| `dead` | ✗ | 빨강 | 강제 삭제 시도가 실패했습니다(예: 장치 사용 중). 컨테이너가 부분적으로만 파괴된 상태입니다. 수동 정리가 필요합니다. |
| `created` | ○ | 회색 | `docker create`는 실행됐지만 `docker start`는 아직 실행되지 않은 상태입니다. |
| `removing` | ○ | 회색 | `docker rm`이 진행 중입니다. 삭제가 완료되면 목록에서 사라집니다. |

### 종료 코드 (die 이벤트에 표시)

| 종료 코드 | 의미 |
|-----------|------|
| `0` | 정상 종료 — 프로세스가 성공적으로 완료됐습니다. |
| `1` | 일반 애플리케이션 오류 — 프로세스 자체가 0이 아닌 상태 코드를 반환했습니다. |
| `130` | `SIGINT` — Ctrl+C를 받은 경우. |
| `137` | `SIGKILL` — 커널(OOM) 또는 `docker kill`에 의해 강제 종료됐습니다. `OOM` 배지를 확인하세요. |
| `143` | `SIGTERM` — `docker stop`이 보낸 정상 종료 신호. |
| 기타 | 애플리케이션이 정의한 종료 코드. 컨테이너 로그에서 원인을 확인하세요. |

## 네트워크 드라이버

| 드라이버 | 설명 |
|---------|------|
| `bridge` | 기본 드라이버. 호스트에 격리된 가상 네트워크를 생성합니다. 같은 브릿지의 컨테이너끼리는 IP로 통신 가능, 외부 트래픽은 NAT 처리됩니다. `--network` 플래그 없이 실행하면 기본 `bridge` 네트워크에 연결됩니다. |
| `host` | 네트워크 격리를 제거합니다. 컨테이너가 호스트의 네트워크 스택을 그대로 공유합니다. 가상 인터페이스나 NAT가 없어 성능이 높지만 호스트 포트가 모두 노출됩니다. |
| `none` | 모든 네트워킹을 비활성화합니다. 루프백(`lo`) 인터페이스만 있습니다. 네트워크 접근을 완전히 차단해야 할 때 사용합니다. |
| `overlay` | 여러 Docker 호스트에 걸친 네트워크(Swarm 모드). 다른 머신의 컨테이너가 같은 가상 네트워크에 있는 것처럼 통신합니다. 단일 호스트 환경에서는 표시되지 않습니다. |
| `macvlan` | 물리 네트워크의 MAC 주소를 컨테이너에 할당해 LAN에서 실제 장치처럼 보이게 합니다. 컨테이너가 회사 네트워크에서 직접 주소 지정이 필요할 때 사용합니다. |
| `ipvlan` | macvlan과 유사하지만 호스트 MAC 주소를 공유합니다. 업스트림 스위치가 포트당 MAC 수를 제한하는 환경에 적합합니다. |

### 시스템 네트워크 (항상 목록 맨 아래 고정)

Docker는 세 개의 네트워크를 자동으로 생성하며, 삭제할 수 없습니다:

- **bridge** — `--network` 미지정 컨테이너의 기본 브릿지
- **host** — 호스트 직접 네트워킹
- **none** — 네트워크 없음

## 이벤트 액션

Events 탭과 Networks 탭 타임라인에 표시됩니다.

| 액션 | 아이콘 | 색상 | 의미 |
|------|--------|------|------|
| `create` | ✦ | 회색 | 컨테이너 객체가 생성됐습니다 (`docker create` 또는 `docker run`의 첫 단계). 프로세스는 아직 실행되지 않습니다. |
| `start` | ● | 초록 | 컨테이너 프로세스가 시작됐습니다. |
| `restart` | ↻ | 파랑 | 컨테이너가 재시작됐습니다 (수동 `docker restart` 또는 재시작 정책). |
| `stop` | ○ | 빨강 | `docker stop`이 `SIGTERM`을 보냈고 유예 시간 내에 정상 종료됐습니다. |
| `kill` | ○ | 빨강 | `docker kill`이 시그널을 보냈습니다 (기본값 `SIGKILL`). 즉시 강제 종료. |
| `die` | ○ | 빨강 | 컨테이너 프로세스가 어떤 이유로든 종료됐습니다 (정상 종료, 충돌, kill 포함). 항상 종료 코드와 함께 표시됩니다. `OOM`은 메모리 한도 초과로 커널이 종료했음을 의미합니다. |
| `pause` | ◑ | 노랑 | `SIGSTOP`으로 모든 프로세스가 일시 정지됐습니다. |
| `unpause` | ● | 초록 | 일시 정지 후 프로세스가 재개됐습니다. |
| `destroy` | ✕ | 빨강 | 컨테이너 객체가 완전히 삭제됐습니다 (`docker rm`). dockviz 목록에서도 제거됩니다. |
| `rename` | • | 회색 | 컨테이너 이름이 변경됐습니다. |
| `exec_create` / `exec_start` | • | 회색 | 컨테이너 내부에서 명령어가 실행됐습니다 (`docker exec`). |
| `health_status` | • | 회색 | 헬스 체크 결과 (healthy / unhealthy / starting). `HEALTHCHECK`가 정의된 경우에만 발생합니다. |

## 이미지 상태

Docker 이미지는 런타임 상태가 없지만, 태그 상태가 중요합니다.

| 상태 | 의미 |
|------|------|
| `이름:태그` | 정상적인 태그된 이미지 (예: `nginx:1.25-alpine`). |
| `<none>` | **댕글링 이미지** — 태그가 없는 레이어. `docker pull`이나 `docker build`로 기존 태그가 새 이미지로 대체될 때 남겨집니다. 디스크 공간을 차지하지만 사용되지 않습니다. `docker image prune`으로 안전하게 정리 가능합니다. |

### 하나의 이미지에 여러 태그

하나의 이미지 ID에 여러 태그가 붙을 수 있습니다 (예: `nginx:latest`와 `nginx:1.25`). dockviz는 태그 하나당 행 하나로 표시합니다. 특정 태그를 삭제하면 그 별칭만 제거되고, 이미지 데이터와 나머지 태그는 마지막 태그가 삭제될 때까지 유지됩니다.

삭제 확인 창에서 다음과 같이 표시됩니다:
- 태그가 여러 개: **"Only this tag will be removed. Image remains (N tags total)."**
- 태그가 하나: **"This will remove the image and free disk space."**

## OOM 강제 종료 (OOM Kill)

컨테이너의 메모리 사용량이 `--memory` 한도에 도달하면 Linux 커널의 OOM Killer가 해당 프로세스를 강제 종료합니다. dockviz에서는 다음과 같이 표시됩니다:

```
○ die   api-server   exit=137  OOM
```

- 종료 코드는 항상 `137` (커널이 보낸 SIGKILL).
- Docker `oomKilled` 속성이 `true`일 때 `OOM` 배지가 표시됩니다.
- **해결책:** `--memory` 한도 증가, 애플리케이션 메모리 사용량 최적화, 또는 스왑 한도 설정.
